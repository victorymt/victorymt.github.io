树递归的两种范式：数学递归 vs 遍历式递归

引言

解决树相关问题时（路径求和、二叉树遍历、哈夫曼编码生成），我们常常陷入一个困惑：到底该用返回值还是用外部容器来保存结果？

答案取决于你的需求——你只想要一个最终统计值，还是需要完整的路径细节。这两种思路分别对应两种递归范式：数学递归（结果计算型）和遍历式递归（过程记录型）。理解它们的区别，能让你写树递归时不再手足无措。

一、两种递归思考范式

1. 数学递归（结果计算型）

核心本质：严格遵循数学递归定义，将大问题拆解为可叠加的子问题，通过子问题的答案组合得到原问题解。

关键特征：
- 基例：最小子问题有明确答案（如叶子节点的路径和就是根到该叶子的累积值）
- 递推关系：大问题答案 = 子问题答案的数学组合（如整棵树路径和 = 左子树路径和 + 右子树路径和）
- 参数特点：仅需核心对象参数（当前节点）和状态传递参数（如已累积的路径和），无需额外结果容器
- 无副作用：函数为纯函数，输入决定输出，不修改外部变量

适用场景：仅需获取最终统计值（如路径总和、哈夫曼树 WPL、节点总数），无需保留路径/编码等细节。

示例（哈夫曼树路径求和）：

class HuffmanNode:
    def __init__(self, freq, left=None, right=None):
        self.freq = freq
        self.left = left
        self.right = right

def sum_path_math(root, current_total=0):
    # 基例1：空节点，子问题答案为0
    if root is None:
        return 0
    # 更新状态参数
    current_total += root.freq
    # 基例2：叶子节点，返回当前路径和
    if root.left is None and root.right is None:
        return current_total
    # 递推关系：合并左右子树的子问题答案
    left_sum = sum_path_math(root.left, current_total)
    right_sum = sum_path_math(root.right, current_total)
    return left_sum + right_sum

2. 遍历式递归（过程记录型）

核心本质：递归仅作为遍历工具，本质上是在完整处理整个问题，通过外部容器收集遍历过程中的细节信息。

关键特征：
- 无明确递推关系：不拆解子问题，而是遍历所有节点/路径
- 结果载体：依赖外部可变容器（列表/字典）保存过程细节
- 参数特点：需要核心对象参数 + 状态传递参数 + 结果容器参数
- 有副作用：函数会修改外部容器的内容

适用场景：需要保留过程细节（如所有路径字符串、哈夫曼编码的字符-路径映射），而非仅获取统计值。

示例（哈夫曼树路径收集）：

def collect_path_traverse(root):
    path_list = []  # 结果容器
    def traverse(node, current_path):
        if node is None:
            return
        # 更新路径状态
        new_path = current_path + str(node.freq) if current_path else str(node.freq)
        # 叶子节点保存路径细节
        if node.left is None and node.right is None:
            path_list.append(new_path)
            return
        traverse(node.left, new_path)
        traverse(node.right, new_path)
    traverse(root, "")
    return path_list

二、两种范式核心对比

对比维度 数学递归（结果计算型） 遍历式递归（过程记录型）
核心逻辑 拆分子问题，组合子问题答案 遍历完整问题，收集过程细节
结果载体 函数返回值 外部可变容器（列表/字典）
思维重点 找「基例」和「递推关系」 找「遍历路径」和「记录条件」
副作用 无（纯函数） 有（修改外部容器）
参数数量 少（核心对象 + 状态参数） 多（核心对象 + 状态参数 + 结果容器）
典型场景 路径总和、WPL计算、节点计数 哈夫曼编码生成、所有路径收集

三、递归路径管理的两种风格

当递归需要记录「路径」时，还有两种具体的实现风格值得对比：

风格 A：路径拷贝 + 递归返回值

每次递归调用前拷贝路径，路径状态是局部的，失败时自然丢弃。

def tracetree(node, target, current_path):
    if node is None:
        return None
    new_path = current_path[:]  # 拷贝路径
    new_path.append(node.val)
    if node.val == target:
        return new_path
    left_result = tracetree(node.left, target, new_path)
    if left_result:
        return left_result
    return tracetree(node.right, target, new_path)
  • ✅ 逻辑清晰，代码简洁
  • ✅ 隐式回溯，无需手动 pop()
  • ❌ 每次递归都拷贝列表，内存开销大

风格 B：共享列表 + 手动回溯

使用一个外部共享列表，所有递归调用修改同一个路径，失败时必须手动撤销。

def get_path_to_max(root):
    max_val = find_max_value(root)
    path = []

    def dfs(node):
        if node is None:
            return False
        path.append(node.val)
        if node.val == max_val:
            return True
        if dfs(node.left) or dfs(node.right):
            return True
        path.pop()  # 手动回溯！
        return False

    dfs(root)
    return path
  • ✅ 避免列表拷贝，更高效
  • ❌ 需要仔细管理回溯逻辑，忘记 pop() 就会出错

四、新手常见错误

1. 混淆递归边界和记录条件

  • ❌ 将空节点（递归边界）当作结果收集点
  • ✅ 空节点仅终止递归，叶子节点才是遍历式递归的记录条件、数学递归的基例

2. 重复累加状态参数

  • ❌ 递归传参时直接 total + root.freq，导致父节点值重复累加
  • ✅ 先就地计算，再将计算后的值传给子树

3. 容器参数未正确修改

  • ❌ 通过 lst + [new_val] 拼接列表（生成新列表，原容器无变化）
  • ✅ 直接调用容器的 append() / update() 方法修改原容器

五、快速选型指南

写递归前问自己两个问题，快速确定使用哪种范式：

  1. 需求是统计值还是过程细节？
  2. 统计值 → 数学递归

  3. 过程细节 → 遍历式递归

  4. 函数核心是计算结果还是执行遍历动作？

  5. 计算结果 → 数学递归
  6. 执行遍历动作 → 遍历式递归
