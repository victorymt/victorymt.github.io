Monad 入门：从 Functor 到 Monad 的直观理解

引言

Monad 是 Haskell 中最让人困惑也最强大的概念之一。很多人第一次接触时，听到的比喻是"盒子"——但 Monad 远不止是一个盒子。

先从直觉开始

从实现上看，Monad 就像一个盒子。放入盒子和从盒子取值都需要特定的操作，经过盒子封装的东西无法通过普通方式访问。

为什么要这样做？Monad 可以用来隔离副作用——把整个流程用盒子封装一遍，让纯函数的世界与外部世界安全地交互。

但 Monad 的含义不止于此，它还有各种各样的组合方式。

Functor → Applicative → Monad

这三者构成了一个递进的抽象层级：

Functor（函子）

Functor 是将一个范畴映射到另一个范畴的结构。

直觉上，当我们把类型看作范畴中的对象、函数看作箭头时，Functor 就是从范畴到范畴之间的映射：

F : C → C

具体到 Haskell 中：
- F(Int) → Maybe Int —— 对应 Haskell 中的 return
- f: Int → Int，则 F(f) : Maybe Int → Maybe Int —— 对应 Haskell 中的 fmap

instance Functor Maybe where
  fmap f (Just a) = Just (f a)
  fmap f Nothing  = Nothing

上面 fmap 的实现利用了柯里化。fmap 的类型签名有两种理解方式：

  1. (Int → Int) → (Maybe Int → Maybe Int) —— 输入一个函数，返回一个函数（函子将态射映射为态射，更符合范畴论视角）
  2. (Int → Int) → Maybe Int → Maybe Int —— 输入一个函数和一个值，返回另一个值（实际的实现）

盒子的操作

从 Functor 到 Applicative 再到 Monad，盒子的能力逐步增强：

抽象 核心操作 能做什么
Functor fmap 把普通函数应用到盒子里的值
Applicative <*> 把盒子里的函数应用到盒子里的值
Monad >>= (bind) 根据盒子里的值决定下一步做什么

Monad 的范畴论意义

从更深的视角看，Monad 可以实现函数的复合：

  • 类型看作范畴中的对象
  • 函数看作箭头
  • 函数的复合看作箭头的复合

当我们处理 Maybe Int、[Int]、IO Int 这类包装类型时，普通的函数复合失效了——你不能直接把一个 Int → Int 的函数应用到 Maybe Int 上。Monad 提供了在"提升后的范畴"中继续复合的能力。

为什么要理解这些？

  1. 隔离副作用：IO Monad 让 Haskell 在纯函数的世界里安全地处理输入输出
  2. 错误处理：Maybe 和 Either Monad 让错误处理变得优雅，无需到处写 try-catch
  3. 状态管理：State Monad 让可变状态在纯函数环境中可控
  4. 异步计算：很多现代语言（JavaScript 的 Promise、Rust 的 Future）本质上都是 Monad
